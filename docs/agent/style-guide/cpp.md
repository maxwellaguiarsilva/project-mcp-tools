# C++ Style Guide

Import: [Common Style Rules](all.md)

The authority chain for C++ formatting in this project:

1. **Templates** — code generated by `cpp_create_class` and `cpp_create_test` defines
   the required pattern. Never alter the automatic boilerplate, even if it seems
   counterintuitive. Observe the structure, code, and comments generated by the
   templates and mimic them.
2. **`cpp_code_verifier`** — conformity is validated through Python and regex checks;
   if the tool reports no errors, the file is in compliance. Violations of spacing
   rules will naturally appear due to formatting patterns inherent to language
   models, so always run the verifier instead of trusting visual inspection.

Compiler context: `g++`, `-std=c++23`, `-Wall -Wextra -Werror -Wfatal-errors`, with
`cppcheck` static analysis via `cpp_analyze`. Warnings are errors.

## Verification Matrix

Rules are enforced at different levels depending on whether they can be
mechanically checked or require semantic understanding:

| Level | Tool | Scope |
|---|---|---|
| **Regex** | `cpp_code_verifier` | Spacing, blank lines, return format, include block layout, and other purely mechanical patterns. See each section's badge below. |
| **Static analysis** | `cpp_analyze` (cppcheck) | Compiler warnings-as-errors, unused variables, uninitialized members, performance traps. |
| **LLM** | Code generation / review | Naming, const-correctness, architectural choices, redundant casts, pipeline style — rules that require understanding intent. |

Individual sections below may be tagged with `[cpp_code_verifier]` (regex),
`[cpp_analyze]` (cppcheck), or `[llm]` (requires judgment) to indicate who owns
the check.

## Case — Lowercase Everywhere `[llm]`

<!-- llm-rule: id=lowercase, complexity=low -->

No capital letters in code files, except in:

- The header with the license and information of the current file.
- Consumption of code and external libraries to this project that require not
  following its standards.
- Scenarios where using lowercase would prevent the code from functioning as
  intended (e.g., case-sensitive language keywords or intrinsic types).
- Markdown files.

```cpp
//	correct
class renderer final : public ::game::renderer

//	incorrect
class Renderer final : public ::game::Renderer
```

## File Anatomy `[cpp_code_verifier]`

Every file starts with the GPL license header (generated by the tools), followed by
the include block, the namespace, the `using` block, and the code. Indentation uses
tabs.

The `cpp_code_verifier` enforces the vertical spacing:

- Maximum of 2 consecutive empty lines anywhere in the file.
- Exactly two empty lines before the first `#include` and after the last one.
- No empty lines between `#include` directives.

```cpp
//	correct: includes grouped, two empty lines around the block
#include <tui/renderer.hpp>
#include <ranges>
#include <sak/using.hpp>


namespace tui {


__using( ::std::, string, vector )
```

```cpp
//	incorrect: empty line splitting the include block
#include <tui/renderer.hpp>

#include <ranges>
```

### Headers

Headers use `#pragma once` plus a numeric header guard, both generated by
`cpp_create_class`:

```cpp
#pragma once
#ifndef header_guard_794521060
#define header_guard_794521060
```

Namespaces are declared as `namespace name {` on the same line, with **no extra
indentation** for their content, and closed with a bare `}` at the end of the file.

## `using` and `__using` `[llm]`

<!-- llm-rule: id=using, complexity=medium -->

Avoid the scope operator `::` at usage sites. Bring symbols in with `using` (single
symbol, tab after the keyword) or the `__using` macro from `sak/using.hpp` (multiple
symbols from the same prefix). `using namespace` is forbidden.

```cpp
//	correct: single symbol
using	::std::to_string;
using	::sak::to_lower_case;
```

The `__using` macro accepts up to 64 parameters. With four or fewer symbols they may
remain on the same line; with more than four, each symbol goes on its own line,
adhering to the "Comma rule":

```cpp
//	correct: four or fewer symbols inline
__using( ::sak::, exit_success, exit_failure, ensure )
__using( ::sak::math::, between )

//	correct: more than four symbols, one per line, leading comma
__using( ::std::
	,lock_guard
	,make_shared
	,unique_lock
	,vector
	,views::iota
	,views::transform
	,views::zip
)
```

Exemptions where `::` is allowed (and required):

- ADL-sensitive STL utilities must stay qualified: `::std::move`, `::std::forward`.
- Accessing nested types within classes.
- Static member calls.
- Avoiding shadowing in exceptional cases of name collision.

```cpp
//	correct: static member call kept qualified
char code	=	::tui::terminal::read_char( );
```

## Naming (C++ specifics)

The common naming rules apply. Additionally:

- Class members prefixed with `m_`, template parameters with `t_`.
- Use descriptive and extensive names.
- Forbidden names:
    - Abbreviations such as `i`, `w`, `h`.
    - Any type-encoding suffix (e.g., `_f`, `_i`, `_t`, `_type`, `_str`, `_ptr`).
- STL, external libraries, and macros with numeric prefixes are exempt.

```cpp
//	correct
for( const auto& pixel : line.start | line_to( line.end ) )
	plot_unsafe( pixel[ width_index ], pixel[ height_index ] );

//	incorrect
for( const auto& p : line.start | line_to( line.end ) )
	plot_unsafe( p[ w ], p[ h ] );
```

## Spacing `[cpp_code_verifier]`

Internal spacing in `( )` and `[ ]` (and brace initializers). The verifier enforces a
space after `(`, `[`, `{` and before `)`, `]`, `}`.

```cpp
//	correct
if( condition )
array[ index ]
ensure( 1 + 1 == 2, "math is broken" );
auto lock = lock_guard( m_mutex );

//	incorrect
if (condition)
array[index]
ensure(1 + 1 == 2, "math is broken");
```

This rule does not apply to string literals. Tokens like formatting placeholders
`{}` must not have internal spaces:

```cpp
//	correct
println( "starting tests for: {}", hierarchy );

//	incorrect
println( "starting tests for: { }", hierarchy );
```

## Commas (C++ specifics)

The common comma rule applies. The constructor initializer list is the canonical
C++ example of leading commas:

```cpp
renderer::renderer( terminal& terminal )
	:m_terminal( terminal )
	,m_margin( { 0, 1 } )
	,m_color( 15 )
	,m_terminal_listener( make_shared< terminal_listener >( *this ) )
{
	m_terminal += m_terminal_listener;
	on_resize( m_terminal.size( ) );
}
```

Enums also follow the leading comma style:

```cpp
enum class direction
{
	 up	=	0
	,down
	,left
	,right
	,none
};
```

## Operators `[llm]`

<!-- llm-rule: id=operators, complexity=low -->

Use the keyword forms: `and`, `or`, `not`, `not_eq`. The symbol forms `&&`, `||`,
`!`, `!=` are forbidden.

```cpp
//	correct
if( main_upper not_eq copy_upper or main_lower not_eq copy_lower )
if( not bounds_start.is_inside( position ) )

//	incorrect
if( main_upper != copy_upper || main_lower != copy_lower )
if( !bounds_start.is_inside( position ) )
```

The pipe operator `|` used in ranges pipelines is exempt and must **not** be
replaced by `bitor`:

```cpp
auto rows = m_main | chunk( m_screen_size[ width_index ] )
	| drop( area.start[ top_index ] )
	| take( area_bound[ height_index ] )
	| transform( crop_width );
```

## Flow Control `[llm]`

<!-- llm-rule: id=flow_control, complexity=low -->

No braces `{}` for single-line `if`, `for`, `while`:

```cpp
//	correct
if( index < m_main.size( ) )
	m_main[ index ] = m_color;

for( auto row : rows )
	fill( row, m_color );

//	incorrect
if( index < m_main.size( ) )
{
	m_main[ index ] = m_color;
}
```

## `return` Spacing `[cpp_code_verifier]`

`return` is separated from its expression by exactly one copy of the whitespace
character that precedes it — a tab in normally indented code. A bare `return;` takes
no separator.

```cpp
//	correct
	return	exit_success;
	return	m_screen_size;
	if( not lock.owns_lock( ) ) return;

//	incorrect
	return exit_success;
```

## Definitions `[llm]`

<!-- llm-rule: id=definitions, complexity=low -->

In `.cpp` files, single-statement methods go on the same line as the signature.
Constructors with initializer lists are the exception (see the "Comma rule").

```cpp
//	correct
auto renderer::set_color( const byte color ) noexcept -> void { m_color = color; }

void renderer::operator +=( const shared_ptr< listener >& subject ) { m_dispatcher += subject; }

game::renderer_listener::renderer_listener( const g2i::point& new_size )
{ on_resize( new_size ); }
```

## Program Termination `[llm]`

<!-- llm-rule: id=termination, complexity=medium -->

Always exit the program by returning an exit code from the `main` function. Use `exit_success` and `exit_failure` from `sak/ensure.hpp` (brought in via `using` or `__using`) as the return values.

Calling `std::exit()` (or `std::quick_exit()`, `std::abort()`) is strictly forbidden, as it bypasses the destruction of automatic (stack-allocated) objects, leading to resource leaks.

```cpp
//	correct
auto main( ) -> int
{
	__using( ::sak::, exit_success, exit_failure )

	//	...

	return	exit_success;
}

//	incorrect: using std::exit is forbidden
auto main( ) -> int
{
	::std::exit( 1 );
}
```

## Console Output `[llm]`

<!-- llm-rule: id=console_output, complexity=low -->

Use `std::println` for all console output. Direct iostream usage (`std::cout`, `std::cerr`, `std::endl`, and `<<` operators) is forbidden in test and application code. This ensures consistent formatting and avoids mixing print styles.

For error output, route through `stderr` via `println( stderr, "..." )`.

```cpp
//	correct
println( "result: {}", value );
println( stderr, "error: {}", error.what( ) );

//	incorrect
cerr << "error: " << error.what( ) << endl;
cout << "result: " << value << endl;
```

### File-level Exception

Files that fundamentally depend on iostream (e.g., wrapping `cout`/`cerr` as `ostream&` references) may be exempted by adding the following comment near the file header:

```
//	#ignore-iostream-rule-file: <justification>
```

This marker signals a deliberate, justified exception to the rule. See `terminal.hpp` and `terminal.cpp` for a canonical example.

## Methods `[llm]`

<!-- llm-rule: id=methods, complexity=medium -->

Mark methods that do not change state with `const` and `noexcept`:

```cpp
auto size( ) const noexcept -> g2i::point override;
```

## Indirection `[llm]`

<!-- llm-rule: id=indirection, complexity=medium -->

- Prefer `const&` (constant reference) to avoid `->` operator noise.
- Adjust the names to avoid collision and shadowing, in order to prevent `this->`.

```cpp
//	correct: reference aliases keep the body free of pointer noise
const auto& listener		=	*m_renderer_listener;
const auto& label_position	=	listener.label_position;

//	correct: parameter named to avoid shadowing the member
explicit terminal_listener( renderer& parent ) : m_renderer( parent ) { }
```

## Increment `[llm]`

<!-- llm-rule: id=increment, complexity=low -->

Prefer pre-increment (`++index`) and pre-decrement (`--index`).

## Comments (C++ specifics)

The common comment rules apply. Additionally:

- Allowed only to explain subtle behaviors or in tests.
- Format: tab after `//`.

```cpp
//	a comment that does not contain any capital letters or periods at the end of the sentence
//	it has a tab \t after // and only one sentence per line
```

## Modernity, Composition, and the `sak` Library `[llm]`

<!-- llm-rule: id=modernity, complexity=high -->

The project favors declarative pipelines over manual loops: `ranges`, `views`, and
custom niebloids in `sak`. Code should be a statement of intent ("what"), not a CPU
instruction manual ("how").

- The `sak` library holds generic, domain-independent utilities (math, geometry,
  patterns). It must not contain business logic or hardware dependencies.
- If logic can be named, it should be a utility in `sak`. Avoid ad-hoc lambdas;
  prefer composing named entities.
- Any lambda with generic, reusable logic that can be expressed via `ranges`,
  `views`, or other `sak` utilities should be implemented as a niebloid or CPO in
  `sak`.

```cpp
//	correct: named composition
for( const auto& pixel : line.start | line_to( line.end ) )
	plot_unsafe( pixel[ width_index ], pixel[ height_index ] );
```

## Casts `[llm]`

<!-- llm-rule: id=casts, complexity=medium -->

Cast only when the language cannot resolve the type implicitly. Redundant casts
are noise — they distract from intent. When the declared type already drives the
conversion, omit the cast.

```cpp
//	correct: no cast needed, int converts implicitly to float
const g2f::point size{ width, height };

//	incorrect: noisy, the type already communicates the intent
const g2f::point size{ static_cast< float >( width ), static_cast< float >( height ) };
```

## Tests `[llm]`

<!-- llm-rule: id=tests, complexity=medium -->

- Use `ensure( bool expression, const string& message )` from `sak/ensure.hpp`
  instead of `cassert`.
- Create tests with `cpp_create_test`: structured tests mirror the source hierarchy
  (`tests/path/test_path_hierarchy.cpp`); adhoc experiments live in
  `tests/adhoc/nnnn_name/`.
- The generated test skeleton (main with `{ }`, try/catch reporting,
  `exit_success`/`exit_failure`) must be kept as-is and filled in.

```cpp
ensure( 1 + 1 == 2, "math is broken" );
```
